# Usage Guide

This guide covers how to run, configure, and connect the Docker-based best-of-N Ollama proxy stack.

## What The Stack Runs

The default Docker setup can run three services:

- `ollama`: local Ollama server in Docker
- `bon-proxy`: OpenAI-compatible best-of-N proxy
- `open-webui`: UI connected to the proxy

The proxy exposes synthetic model IDs such as:

- `bon/llama3.1:8b`
- `bon/qwen2.5:14b`
- `bon-pool/coding`

When Open WebUI sends a chat request to one of these models, the proxy:

1. Generates `N` candidates from Ollama
2. Asks the configured judge model to choose a winner
3. Returns only the selected answer to Open WebUI

## Files You Will Edit

For the managed Docker workflow, use one main file:

- `config/stack-config.json`

Start from:

```bash
cp config/stack-config.json.example config/stack-config.json
```

This file has two sections:

- `stack`: Docker-facing settings like ports and the API key
- `proxy`: best-of-N proxy behavior and model definitions

You normally do not need to edit `.docker/generated.env` or `.docker/proxy-config.json` directly. They are generated by `stack.sh`.

## Quick Start

1. Create your config:

```bash
cp config/stack-config.json.example config/stack-config.json
```

2. Review and edit `config/stack-config.json`.

3. Start the stack:

```bash
bash stack.sh start
```

4. Pull the models referenced by your config:

```bash
bash stack.sh pull-models
```

5. Open Open WebUI:

```text
http://localhost:3000
```

## Stack Commands

Use [stack.sh](/home/chris/council/stack.sh) for normal operations.

```bash
bash stack.sh start
bash stack.sh stop
bash stack.sh restart
bash stack.sh status
bash stack.sh logs
bash stack.sh config
bash stack.sh pull-models
```

Command summary:

- `start`: builds and starts Docker services
- `stop`: stops and removes containers
- `restart`: recreates the stack
- `status`: shows running containers
- `logs`: tails Docker logs
- `config`: regenerates derived config files without starting services
- `pull-models`: pulls all configured judge/base/pool models into the Ollama container

## Configuration Reference

Example structure:

```json
{
  "stack": {
    "host_ollama_port": 11434,
    "host_proxy_port": 8000,
    "host_open_webui_port": 3000,
    "open_webui_api_key": "dummy"
  },
  "proxy": {
    "judge_model": "qwen2.5:7b-instruct",
    "ollama_base_url": "http://ollama:11434",
    "default_n": 3,
    "request_timeout_seconds": 120,
    "judge_retry_count": 1,
    "passthrough_models": true,
    "models": {},
    "pools": {}
  }
}
```

### `stack`

These settings affect Docker wiring and host access.

- `host_ollama_port`: host port for the Ollama container
- `host_proxy_port`: host port for the proxy API
- `host_open_webui_port`: host port for Open WebUI
- `open_webui_api_key`: API key used by Open WebUI when calling the proxy

### `proxy`

These settings control the proxy behavior.

- `judge_model`: Ollama model used to rank candidates
- `ollama_base_url`: upstream Ollama URL used by the proxy
- `default_n`: default number of candidate generations
- `request_timeout_seconds`: upstream request timeout
- `judge_retry_count`: how many times to retry judging after invalid judge output
- `passthrough_models`: whether to also expose upstream Ollama models directly
- `models`: synthetic model definitions
- `pools`: named candidate pools for multi-model best-of-N

### Synthetic Model Definitions

Each entry in `proxy.models` defines one model ID exposed to Open WebUI.

Single-model best-of-N example:

```json
"bon/llama3.1:8b": {
  "base_model": "llama3.1:8b",
  "n": 3,
  "sampling": {
    "temperature": 0.8,
    "top_p": 0.95
  }
}
```

Fields:

- `base_model`: Ollama model to sample repeatedly
- `n`: number of candidates
- `sampling.temperature`: optional override for temperature
- `sampling.top_p`: optional override for top-p
- `sampling.max_tokens`: optional override for max tokens

### Pool Definitions

Pools let one synthetic model draw candidates from multiple underlying models.

Example:

```json
"bon-pool/coding": {
  "pool": "coding",
  "n": 3
}
```

```json
"coding": {
  "candidates": [
    {
      "model": "qwen2.5-coder:14b",
      "count": 2
    },
    {
      "model": "deepseek-coder:6.7b",
      "weight": 1
    }
  ]
}
```

Candidate fields:

- `model`: underlying Ollama model
- `count`: fixed number of slots in the pool
- `weight`: proportional share of remaining slots
- `sampling`: optional per-candidate sampling overrides

## Local Ollama vs Network Ollama

The important setting is:

```json
"proxy": {
  "ollama_base_url": "..."
}
```

Choose one of the following modes.

### Mode 1: Local Ollama In This Docker Stack

This is the default setup.

Use:

```json
"ollama_base_url": "http://ollama:11434"
```

Behavior:

- Docker starts its own `ollama` container
- The proxy talks to it over the Docker network
- `bash stack.sh pull-models` pulls models into that container

Use this mode if you want the entire stack self-contained in Docker.

### Mode 2: Local Ollama Running On Your Host Machine

If you already run Ollama directly on your machine outside Docker, point the proxy at the host.

On Linux with Docker host-gateway support, the most common value is:

```json
"ollama_base_url": "http://host.docker.internal:11434"
```

If your Docker environment does not resolve `host.docker.internal`, use your host LAN IP or another reachable host address instead:

```json
"ollama_base_url": "http://192.168.1.50:11434"
```

Behavior:

- The proxy container uses your existing host Ollama server
- The Docker `ollama` container may still start if you use `bash stack.sh start`
- In this mode, you can ignore the Docker Ollama container or remove it from Compose later if you want a leaner setup

Important:

- Your host Ollama server must listen on an address reachable from Docker
- A localhost-only bind on the host may not be reachable from the proxy container

### Mode 3: Ollama On Another Machine On Your Network

Point the proxy at the remote Ollama server:

```json
"ollama_base_url": "http://10.0.0.25:11434"
```

Behavior:

- The proxy uses an Ollama server on another machine
- The remote machine must allow inbound access on the configured port
- Models must exist on that remote Ollama instance

Checklist for network Ollama:

- Confirm the Ollama server is reachable from the Docker host
- Confirm the Ollama port is open
- Confirm firewall rules allow access
- Confirm the required judge/base/pool models are installed remotely

### Which Mode Should You Choose?

Use local Docker Ollama if:

- You want the simplest all-in-one setup
- You want `stack.sh pull-models` to manage model downloads

Use host Ollama if:

- You already have a tuned local Ollama instance
- You want to reuse existing downloaded models on the host

Use network Ollama if:

- You want to offload inference to another machine
- You already have a GPU server running Ollama elsewhere

## Open WebUI Integration

### If You Use The Included Docker Stack

Open WebUI is already configured to call the proxy internally.

Open the UI at:

```text
http://localhost:3000
```

The UI should see the proxy-exposed model list from `/v1/models`.

### If You Run Open WebUI Separately

Add the proxy as an OpenAI-compatible provider with:

- Base URL: `http://localhost:8000/v1`
- API key: value from `stack.open_webui_api_key`

If Open WebUI runs on another machine, use the Docker host IP instead of `localhost`.

## Common Tasks

### Change The Number Of Candidates

Global default:

```json
"default_n": 3
```

Per model:

```json
"bon/qwen2.5:14b": {
  "base_model": "qwen2.5:14b",
  "n": 5
}
```

### Change The Judge Model

```json
"judge_model": "llama3.1:8b"
```

Then pull it:

```bash
bash stack.sh pull-models
```

### Add A New Synthetic Model

Example:

```json
"bon/mistral": {
  "base_model": "mistral:7b",
  "n": 4,
  "sampling": {
    "temperature": 0.9
  }
}
```

### Add A New Pool

Example:

```json
"models": {
  "bon-pool/general": {
    "pool": "general",
    "n": 4
  }
},
"pools": {
  "general": {
    "candidates": [
      { "model": "llama3.1:8b", "count": 2 },
      { "model": "qwen2.5:14b", "count": 1 },
      { "model": "mistral:7b", "weight": 1 }
    ]
  }
}
```

## Troubleshooting

### Open WebUI Shows No Models

- Confirm the proxy is running on port `8000`
- Confirm `bon-proxy` is healthy in Docker logs
- Confirm `config/stack-config.json` defines models under `proxy.models`
- Confirm Open WebUI is pointed at the proxy, not directly at Ollama

### Proxy Cannot Reach Ollama

- Check `proxy.ollama_base_url`
- If using host Ollama, verify the address is reachable from Docker
- If using network Ollama, verify firewall and routing
- Confirm Ollama is serving on port `11434` or your chosen port

### Judge Fails Or Responses Fall Back Often

- Make sure the judge model is installed
- Try a more capable judge model
- Increase `request_timeout_seconds`
- Review proxy logs for `judge_selected` and `fallback_selected` events

### `pull-models` Does Not Affect A Remote Ollama

That is expected. `bash stack.sh pull-models` pulls models into the Docker `ollama` container only. If you use host or network Ollama, install models on that Ollama instance directly.
